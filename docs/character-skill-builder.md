# 人物 Skill 资产生成说明

## 1. 文档定位

本文档说明 `StoneStory-skill` 中人物 skill 资产是如何由输入数据构建出来的。

本文档关注：

- 人物资产生成模块的目标与边界
- 输入文件、输出文件与代码位置
- 当前证据分层链路如何工作
- 当前实现的限制与后续优化方向

本文档不负责：

- 角色对话运行时如何消费这些资产
- 人物回答生成模块如何调用模型
- 评估模块如何给回答打分

相关文档：

- 数据构建说明：`docs/data-builder.md`
- 人物对话实现说明：`docs/character-chat-implementation.md`
- 人物对话路线图：`docs/character-chat-plan.md`

## 2. 模块目标

人物 skill 资产生成模块的目标，不是直接自动写出完整人物画像，而是先建立一条“证据优先”的人物资产构建链路。

当前模块主要解决：

- 从关系数据中整理人物直接关系
- 从原文段落中抽取人物相关证据
- 对证据做去噪与相关性排序
- 从证据中提炼更适合分析人物说话风格的子集
- 生成可回溯的风格候选结论

当前模块明确不做：

- 直接写无证据支撑的人物性格判断
- 直接生成最终角色口吻
- 直接生成正式回答结果

## 3. 代码位置

- `main_character_skill.py`

核心代码：

## 4. 输入与输出

### 4.1 输入

当前主要输入文件：

- `skill/characters/*/*.skill/manifest.json`
- `data/input/triples.csv`
- `data/output/passages.jsonl`

各输入职责如下：

- `manifest.json`
  - 提供人物基础标识、别名、证据包含词、排除词
- `triples.csv`
  - 提供人物直接关系三元组
- `passages.jsonl`
  - 提供原文段落级证据候选

### 4.2 输出

当前每个人物目录下会生成或更新下列文件：

- `relations.md`
- `evidence_passages.jsonl`
- `evidence_ranked.jsonl`
- `style_evidence.jsonl`
- `style_summary_candidates.json`
- `source_report.json`
- `persona.md`
- `style.md`
- `examples.md`

其中主要文件职责如下：

- `relations.md`
  - 记录可确认的人物直接关系
- `evidence_passages.jsonl`
  - 记录人物相关的原始证据候选
- `evidence_ranked.jsonl`
  - 记录带打分和噪声标记的证据
- `style_evidence.jsonl`
  - 记录更适合分析人物表达风格的证据子集
- `style_summary_candidates.json`
  - 记录带证据支撑的结构化风格候选
- `source_report.json`
  - 记录本次生成过程的计数、来源与统计信息
- `persona.md`
  - 记录人物基础事实与约束
- `style.md`
  - 记录基于证据候选整理出的风格结论
- `examples.md`
  - 记录示例或示例生成约束

## 5. 当前证据分层链路

当前人物资产生成不是一步完成，而是分为四层。

## 5.1 第一层：原始证据召回

输出文件：

- `evidence_passages.jsonl`

核心做法：

- 逐条扫描 `passages.jsonl`
- 使用 `manifest.json` 中的 `evidence_terms` 匹配段落
- 若只命中 `exclude_terms` 对应短语，则视为误命中，不收录

这一层回答的问题是：

- 哪些段落与该人物直接相关

这一层不回答：

- 哪些段落质量更高
- 哪些段落更适合分析说话风格

## 5.2 第二层：去噪与相关性排序

输出文件：

- `evidence_ranked.jsonl`

核心做法：

- 对原始证据逐条打分
- 记录增强信号 `signals`
- 记录噪声标记 `noise_flags`

当前使用的增强信号主要包括：

- 命中人物全名
- 命中别名
- 同时命中多个识别词
- 包含引语符号
- 包含目标人物发话标记
- 包含人物在场信号

当前使用的噪声标记主要包括：

- 只命中短别名
- 无对话信号
- 包含易混淆对象名
- 目标人物未被明确识别为说话人
- 多个发话人候选并存
- 泛指发话主体场景

这一层回答的问题是：

- 哪些人物相关段落更值得优先关注
- 哪些段落虽然相关，但质量较弱或噪声较大

当前这一层仍然是“打分排序”，不是“硬过滤”。

## 5.3 第三层：风格证据提炼

输出文件：

- `style_evidence.jsonl`

核心做法：

- 从第二层结果中继续筛选
- 优先保留明确对话上下文中的段落
- 优先保留目标人物是明确说话人的段落
- 要求证据达到一定相关性分数

这一层回答的问题是：

- 哪些段落适合拿来分析人物如何说话
- 哪些段落适合作为 `style.md` 的直接依据

这一层仍不直接做：

- 直接写人物风格结论
- 直接生成正式回答样例

## 5.4 第四层：风格候选归纳

输出文件：

- `style_summary_candidates.json`

核心做法：

- 从 `style_evidence.jsonl` 中提取更稳定的风格模式
- 生成保守的结构化候选，而不是直接写成长段画像

每条候选当前绑定：

- `trait`
- `title`
- `description`
- `rationale`
- `evidence_passage_ids`
- `evidence_chapter_ids`
- `signal_count`

这一层回答的问题是：

- 如何把风格证据整理成可复用的结论候选
- 如何让 `style.md` 从说明性文件升级成证据化资产

## 6. 为什么要做分层

人物资产生成里，下面几件事并不等价：

- 人物相关
- 人物相关且质量较高
- 适合做风格分析
- 适合沉淀成稳定风格结论

因此分层的目的，是避免把“只提到人物名字”的段落直接拿去写人物风格，也避免人物设定重新退回到主观印象总结。

可以简化理解为：

- 第一层回答“有没有关系”
- 第二层回答“证据质量如何”
- 第三层回答“能不能拿来研究说话风格”
- 第四层回答“能不能沉淀成可回溯的风格结论”

## 7. 与人物目录中其他文件的关系

当前人物目录中的几个核心文件可这样理解：

- `relations.md`
  - 关系事实文件
- `persona.md`
  - 数据约束版人物说明
- `style.md`
  - 证据化风格文档
- `examples.md`
  - 示例或示例生成约束文件

当前这一套设计的重点是：

- 先把关系事实和原文证据整理出来
- 再把更可靠的风格证据筛出来
- 最后再沉淀成可回查的风格结论

## 8. 当前实现限制

当前实现仍有这些限制：

- 原始证据数量仍偏多，去噪主要依赖排序而不是正式过滤
- 风格证据筛选仍以规则为主，不是完整的人物语言建模
- 证据相关性主要依赖字符串匹配与规则信号
- `triples.csv` 中若存在脏数据，会直接影响关系输出

## 9. 后续优化方向

后续优化建议沿下面几个方向推进：

1. 为 `evidence_ranked.jsonl` 引入更稳定的正式过滤阈值
2. 增加更细的说话人识别规则，继续减少“人物在场但并未发话”的误收录
3. 为风格证据增加更细标签
4. 提高风格候选的区分度与稳定性
5. 基于风格候选生成更可信的示例与回答约束

## 10. 关于优化方案的当前结论

当前实现采用的是规则驱动 baseline。

这里的 baseline 指的是：

- 先用可解释、可复现、低依赖的方案把链路跑通

当前可以成立的结论是：

- 规则方案已经能支撑第一版人物资产生成
- 当前链路已经实现从关系事实到原文证据、再到风格候选的分层构建

当前不能直接下结论的是：

- 规则方案一定优于 embedding 方案
- 模型打分一定优于规则方案
- 当前风格证据已经足够支撑最终人物口吻生成

因此，embedding 检索和模型重排目前应视为后续可验证的优化方向，而不是已经完成的能力。
